# recovery_bot — повторяемый анализ live trades vs backtest trades

Цель документа: зафиксировать повторяемый анализ для `BTC 5m recovery`, где мы сравниваем на **одном и том же периоде** и на **одних и тех же рынках**:

- `real live WR` по фактическим real-ставкам бота
- `live trade-based WR` по `market_trade_history` из `data/recovery_bot.db`
- `backtest trade-based WR` по `pm_trades` из `data/backtest.db`

Текущий скрипт для этого анализа:

```bash
venv/bin/python scripts/recovery_live_vs_backtest_analysis.py
```

Быстрый режим без сетевой догрузки historical-рынков:

```bash
venv/bin/python scripts/recovery_live_vs_backtest_analysis.py --no-sync-backtest
```

Текущая версия скрипта считает по всем `BTC 5m` рынкам, где вообще есть `market_trade_history`.

Изначально документ описывал более строгий вариант только по полностью записанным рынкам. Этот strict-фильтр сейчас отключён, потому что он слишком сильно резал выборку и искажал сравнение.

## Что именно сравниваем

Стратегия фиксированная, только `BTC 5m`, только основной first-signal, без repeat:

- `bottom=0.38`
- `entry=0.65`
- `top=0.68`
- `confirm_delay=0.2`
- `confirm_min=0.60`
- `min_seconds_to_expiry=20`
- `max_seconds_to_expiry=240`

Для всех трёх сравнений должна использоваться **одна и та же** логика сигнала:

1. Ждём `touch <= 0.38`
2. После touch ждём первое восстановление в зону `[0.65, 0.68]`
3. Если до этого был `overshoot > 0.68`, цикл сбрасывается
4. Проверяем confirm через `0.2s`
5. Если confirm `< 0.60`, сигнал отбрасывается
6. Берём только `first-signal` по рынку

## Окно анализа

Окно анализа начинается с момента, когда в live была включена запись `market_trade_history`.

Дальше анализ идёт не по всем рынкам подряд, а только по тем `BTC 5m` рынкам, для которых live trade history записана полностью.

## Строгий фильтр (отключён)

Ниже оставлен прежний критерий для справки, но текущий скрипт его не применяет.

Рынок считался бы полным только если в `data/recovery_bot.db`:

- есть `market_trade_history` по этому `market_id`
- запись начинается достаточно рано, чтобы не пропустить старт рынка
- запись продолжается до конца рынка
- нет явного обрезания окна в начале или в конце

Практический критерий для автоматизации:

1. Берём `market_start` и `market_end` рынка
2. Смотрим `min(ts)` и `max(ts)` в `market_trade_history` по этому `market_id`
3. Рынок считаем полным, если:
- `min(ts) <= market_start + допустимый_буфер`
- `max(ts) >= market_end - допустимый_буфер`

Текущий рабочий критерий:

- начало: `10s`
- конец: `10s`

То есть рынок считается полностью записанным только если live trade history покрывает старт и экспирацию почти без дыр. Буферы нужны только как защита от технической задержки записи на несколько секунд.

## Что должен делать повторяемый анализ

### 1. Найти eligible live-рынки

Из `data/recovery_bot.db` выбрать только:

- `symbol='BTC'`
- `interval_minutes=5`
- рынки после включения `market_trade_history`
- рынки с полной live trade history по критерию выше

Это и есть базовая выборка рынков для анализа.

### 2. Проверить покрытие в backtest.db

Для каждого рынка из базовой выборки проверить в `data/backtest.db`:

- есть ли запись в `markets`
- есть ли `condition_id`
- есть ли `pm_trades`

### 3. Догрузить недостающие рынки

Если рынок из live-выборки отсутствует в `backtest.db` или у него нет `pm_trades`, анализ должен:

- добавить рынок в `markets`
- дозаполнить `condition_id`, если он пустой
- скачать `pm_trades` через historical `data-api.polymarket.com/trades`
- сохранить их в `backtest.db`

Важно:

- использовать тот же источник и тот же способ загрузки, что и для обычного historical backtest
- не смешивать live websocket trades с `backtest.db`
- `backtest.db` всегда должен содержать именно historical `pm_trades`

Именно это по умолчанию делает `scripts/recovery_live_vs_backtest_analysis.py`, если не передан `--no-sync-backtest`.

### 4. Посчитать live trade-based WR

На рынках из базовой выборки прогнать стратегию по `market_trade_history` из `data/recovery_bot.db`.

Результат:

- число сигналов
- число wins
- `WR`

### 5. Посчитать backtest trade-based WR

На **тех же самых рынках** прогнать ту же стратегию по `pm_trades` из `data/backtest.db`.

Результат:

- число сигналов
- число wins
- `WR`

### 6. Посчитать real live WR

На тех же рынках взять из `data/recovery_bot.db` реальные ставки бота:

- только `mode='real'`
- только реально выставленные и заполненные сделки
- только по рынкам из базовой выборки

Результат:

- число реальных сделок
- число wins
- `WR`

## Что должно быть в итоговом отчёте

Отчёт должен показывать:

- период анализа
- когда началась запись `market_trade_history`
- сколько `BTC 5m` рынков признаны полными
- сколько рынков пришлось догрузить в `backtest.db`
- `live trade-based WR`
- `backtest trade-based WR`
- `real live WR`

Отдельно нужен breakdown по составу сигналов:

- `same signal`: обе базы дали один и тот же side на одном рынке
- `live-only`: сигнал есть только в `market_trade_history`
- `backtest-only`: сигнал есть только в `pm_trades`
- `different side`: обе базы дали сигнал, но side разный

Для каждой группы нужно показывать:

- число рынков
- число wins
- `WR`

## Что важно сравнивать дополнительно

Помимо общего `WR`, анализ должен отдельно фиксировать:

- сколько рынков есть в общей выборке
- сколько рынков покрыты и live, и backtest
- сколько сигналов теряется в `backtest` относительно live
- сколько сигналов теряется в live относительно `backtest`
- сколько раз расходится `first-side`

Это нужно, потому что основная проблема может быть не только в общем винрейте, но и в том, что два источника вообще выбирают разные first-signal.

## Как запускать периодически

Анализ должен быть идемпотентным и пригодным для регулярного прогона.

При каждом новом запуске он должен:

1. Найти новые live-рынки с полной trade history
2. Сверить их с `backtest.db`
3. Догрузить только недостающее
4. Пересчитать все три `WR`
5. Показать delta относительно прошлого запуска

То есть это не одноразовая ручная аналитика, а повторяемый pipeline сравнения:

- `real live`
- `live trades`
- `historical backtest trades`

на одном и том же срезе рынков.

## Зачем это нужно

Этот анализ должен ответить на главный вопрос:

почему `real live WR` расходится с `backtest WR`.

Для этого нужно развести три возможных источника расхождения:

- бот реально торговал хуже, чем давали бы live trades
- historical `pm_trades` дают не тот же сигнал, что live `market_trade_history`
- проблема не в данных, а в реальном live-исполнении / выборе стороны / missing signals

Без такого анализа сравнение "весь live против всего backtest" малоинформативно, потому что там смешиваются:

- разные периоды
- разные наборы рынков
- рынки с неполной live trade history
